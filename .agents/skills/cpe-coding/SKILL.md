# Coding Style

## エラーの細かい表現は最小限にする
ドメインロジックとして当たり前に発生しうるエラーについては丁寧にハンドリングし、ログ出力、palette上でのメッセージ表示を行うべきだが、めったに起こらないエラー (本当に例外的なケース) についていちいち丁寧にハンドリングする必要はない。
エラーが起こったらまとめて `catch` して `エラーが発生しました。詳細: ${e}` のような感じでシンプルに出せば良い。

## 関数への過度な切り出しをやめる
数行の簡単な処理を、意味づけ (naming) やローカルスコープ汚染を防ぐためだけに、関数に切り出すのには慎重になるべき。
- 簡単な処理 (単にDOM要素をサクッと取得するだとか、ちょっとしたバリデーションのための数行の処理) をいちいち関数に切り出すのはあまりよくない。
- 大抵の場合、取得ロジックの結果を格納する変数名を適切な意味が伝わるように命名すれば、関数への切り出しは不要であることが多い。
- ローカルスコープの変数汚染を避けるには、即時関数式を使うとよい。複数行にわたってのデータ整形などを行う場合にも、即時関数式を用いて変数スコープを狭くすれば大抵の場合事足りる。なお、Promiseオブジェクトがある場合は無理に `(await (async () => {` のように書かずに、シンプルに `then` のメソッドチェーンを使えば良い。
- メインロジックを簡潔にするためだけに関数に切り出しすぎるデメリットは、その関数だけを一目見たときに意図がわからないことである。「確かにこの関数の責務は、命名通りに理解できるけれども、そもそもなんでこの処理が必要なのかがわからない」という状況になりかねない。関数への切り出しは、文脈を失わせるというデメリットが有る。
  メインの処理の中での細かい処理は、文脈を失ってでも共通化するべきときにのみ関数に切り出すべきだ。
  > 拡張機能開発では、通常のWebアプリ開発 (特にバックエンド) と違って、DB呼び出し、外部サービス呼び出しのような、呼び出し背景がわかりやすい処理ばかりではなく、Chromeランタイムに深く結びついた処理が多い。それを単に関数化して関数名でラベリングするだけでは、その処理が必要になる背景が見えづらい。

## ファイルやモジュールの単位で意味的に凝集化する
AIが提出するコードでは、必要以上に関数や切り出した定数をエクスポートして、その都度外部に知識が漏れ出ていることがしばしばある。
- マジックナンバーを切り出して外部に使わせる場合、それは判定ロジックを外部に流出させることになる。
- マジックナンバーや関数を個々の単位でいくつもexportするような状況は、設計段階で凝集化やレイヤー化、疎結合化、単一責任化等の観点から修正を検討すべきシグナルである。
- たとえば、I/Oするためのストレージ読み出しキーをマジックナンバー化して切り出すと、外部に大量のストレージI/O処理が独自実装されることになり管理できなくなる。  
  そうした場合に、ストレージI/OロジックをDAO (データアクセスを単に抽象化してDTOの受け渡し入出力をする場合) や Repository (DTOだけでなく、メソッド等の振る舞いを持ちロジックをカプセル化したオブジェクト単位で入出力する場合) 等を用いることで、I/Oの窓口のみを外部に露出し、マジックナンバーは隠蔽することができる。
- たとえば、キーバインドや特定のキーワードで発火する処理を、単に `const KEY_NAME` 等を用いて切り出す状況は、単に１個の機能を実装するだけなら十分かもしれないが、こちらも「発火」の部分と詳細のドメインロジック等を混同した状況といえる。  
  専用のrouter, dispatcher, launcher等のコアモジュールを用意して、そこにregisterする形式で実装すると、関心の境界を明確に分け、ロジックの凝集化を助ける。


## 特定Webアプリに依存した処理では、マジックナンバーの過度な定数切り出しをやめる
コアロジック、コマンドパレットのフレームワーク側で重要なコードにおいては、コアロジックとマジックナンバーを分離することに大きなメリットがあるため採用して良い。
しかし、ほとんどのwebアプリのオートメーションのユースケースでは、どこをどう切り取ってもマジックナンバーばかりで、外部性を見かけるたびに外に切り出していたら、キリがない。
むしろ外部性に密着してロジックを構築する方が自然である。
関数への切り出しをやめる、のセクションに説明したのと同様に、定数値の外部切り出しには「文脈が失われる」「意図・存在理由がわからない」というデメリットが有るからだ。

## グローバル依存・副作用は減らす・明示する
一部のロジックを切り出した関数において、その内部ではできる限りグローバルコンテキストに依存した処理や副作用を起こす処理を書かない。
そうした処理は、次のような方法によってより安全に記述することができる
- 外部に依存する部分は、呼び出し元から引数として値で受け取る
- 外部に依存する部分の複雑な呼び出しや、副作用を起こすための部分は、コールバック関数で受け取る
- classで書き換え、外部への依存はコンストラクタのdependency injectionによって明示する
- そもそも、ロジックを切り出した関数が外部への依存・副作用をもつのはおかしいと考え、boundaryを再検討する

ただし、各コマンド等のエントリポイントを担う関数においては、そうした副作用を含む様々なコンテキストを注入する必要があるので、疎結合化・純粋関数化に努める必要はない。

## フレームワーク依存はできるだけ切り分ける
たとえば、純粋にロジック本体を書いている部分で React や Solid.jsの特有の機能 (useState, createResource等) を使うのは、ロジックの純粋性を損なうことになるため、避けるべき。フロントエンド技術都合とロジックは分ける。
一方の、Chrome拡張のランタイムAPIに関しては、ロジックとの切り分けが難しいケースもある。
たとえば、どうあがいても chromeランタイムの様々な仕様は全体のアーキテクチャに影響を与えてしまう。
そういうケースについては、もちろん理想的なのはできる限り抽象化や疎結合化・DIP等を行うことではあるが、現実的にはそれをせずトランザクションスクリプト的に書いたほうが見通しが良くなることも多い。前章のグローバル依存・副作用を減らすテクニックは使いつつも、コード量が増えすぎない・かえって見栄えがわるくなったりしない程度のバランスを取ろう。

## リンターについて

anyを使ったり、eslint-disableを使ったりするのは基本的に離れ業だ。
もちろんやむを得ず使わないといけないケースや、そもそもESLintルール整備が不十分なケースもあるので、勝手にanyやdisableを使って進むのではなく、必ずユーザーに許可を得ること。
なお、anyはそもそもほとんどのケースでunknownで十分なことも多い。anyを使いたくてeslint-disableしたくなったら、まずはunknownを使うことを検討する。

## as constの使用には少し慎重になること

as constは非常に協力なツールであり、typeofやsatisfiesとも組み合わせて高度な演算が可能である。
しかし、たとえば不用意に列挙型の用途で使ったり (※列挙型は使わず、基本的には文字列リテラルのユニオン等が望ましい) 設計次第で不必要になるところまで適用するのは避けたい。
as constを使うべきところは、as constでなければ課題解決できないところに限るべきで、as constなしで実現できる方法を一度は考えるべきである。
とはいえ、as constを避けるために設計を歪めたり型安全性・ランタイム安全性を犠牲にして良いわけではなく、まさにそういうときこそas constで突破すべきなのである。

## YAGNI実装をしない
将来必要になるかも知れない、という理由で、不用意に引数のパターンを増やしたり、安定したデフォルト値があるのに変更可能にしたりしない。

## コメントアウトでは、説明をしすぎない
AIエージェントはコメントを書くのが好きなようで、コメントアウトで５行も１０行も説明を書くことが多い。
それらが意味のあるコメントであり、初見の人にとってもそのシンボルの理解に必要十分なものであればよいのだが、無駄であることのほうが実際には多い。
- 「このシンボルは、◯◯の箇所で呼び出す」は基本的にはNG。呼び出し先は、コードの静的解析ですぐ追えるから。どこで呼ばれるか、ではなく、なぜこのシンボルや処理が存在するのか、このシンボルや処理は何をする責務を持つのか、といったことに集中して説明したい
- AIエージェントとユーザーによる会話の中で出てきた修正のための指示を、そのままコメントに書き写すのもNG。たとえばユーザーが「マジックナンバー切り出しはやめてください。」と指示したときに、その内容をそのままコメントに「// マジックナンバー切り出しを避ける。」のように書き写すのは、価値がない。ユーザーが指示したのは、その実装時においてあまりよくない実装パターンが出てきてしまったから、そのnegativeをneutralに戻すためである。こうした経緯は、Architecture Decision等の重要な意思決定・変更経緯のように情報として維持される必要がまったくないのだ。

## 各所の `lib` や `core` ディレクトリの資産を活用する (RPC, DOM操作)

### 実行コンテキストをまたいだ処理にはJSON-RPC風ライブラリを利用する。
実行コンテキストをまたいで処理を行いたい場合は、JSON RPC風の軽量RPCライブラリを自前で実装しているので、それを活用すること。
なお、一般的にはRPCはスキーマと実装が別になっており実装テンプレートが自動生成される事が多いが、今回のケースでは実行コンテキストを全て同一のコードベース上で管理できるため、「関数で実装し、名前をつけて RpcRoute (src/pages/lib/rpc/types.ts) としてエクスポートする」という形式で良い。実装ファーストのスキーマ管理 (さらに、コードベース上も厳格な分離を強制しない) である。

### DOM操作には、非同期セレクタや仮想マウスクリックのライブラリを使用する
また、DOM操作について、`waitForSelector` のように汎用的にDOM要素の監視をする仕組みや `simulateMouseClick` のように単なるclick()呼び出しよりも人間の操作に近い形でシミュレートする仕組みなども活用すること。
