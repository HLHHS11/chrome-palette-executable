# Coding Style

## エラーの細かい表現は最小限にする
ドメインロジックとして当たり前に発生しうるエラーについては丁寧にハンドリングし、ログ出力、palette上でのメッセージ表示を行うべきだが、めったに起こらないエラー (本当に例外的なケース) についていちいち丁寧にハンドリングする必要はない。
エラーが起こったらまとめて `catch` して `エラーが発生しました。詳細: ${e}` のような感じでシンプルに出せば良い。

## 関数への過度な切り出しをやめる
単にDOM要素をサクッと取得するだとか、ちょっとしたバリデーションのためにいちいち関数に切り出して表現するのは避けるべき。
大抵の場合、取得ロジックの結果を格納する変数名を適切な意味が伝わるように命名すれば、関数への切り出しは不要であることが多い。あるいは、ローカルで複数行にわたってのデータ整形などを行う場合にも、即時関数式を用いて変数スコープを狭くすれば大抵の場合事足りる。
関数に切り出すと、メインのロジック本体は読みやすくなるかもしれないが、その関数だけを一目見たときに意図がわからない。
関数への切り出しは、文脈を失わせるというデメリットが有る。
メインの処理の中での細かい処理は、文脈を失ってでも共通化するべきときにのみ関数に切り出すべきだ。
